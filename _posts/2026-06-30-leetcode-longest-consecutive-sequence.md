## 发现问题

LeetCode 128 题要求 O(n) 找最长连续序列。排序谁都会，O(n log n) 轻松过——但题目偏不让你排序。那问题来了：**不用排序，怎么判断「连续」？** 更诡异的是，代码里明明有嵌套循环（for 里套 while），为什么复杂度不是 O(n²) 而是 O(n)？

## 骨架认识

这道题的骨架不是「怎么找连续」，而是「**怎么避免重复劳动**」。

```
暴力：每个数都往后数 → 每个数都可能被多数次 → O(n²)
优化：只从「起点」往后数 → 每个数只被数一次 → O(n)
```

**起点的定义**：`num - 1` 不在集合中。就像排队时你不是站在队伍中间开始数人头，而是找到排头再数——排头只有一个，队伍不会重复数。

核心工具是哈希集合（set），它把 `in` 操作从 O(n) 压到 O(1)——这是 O(n²) 变 O(n) 的前提。

## 点 → 线 → 面

### 点 1：为什么必须用 set？

```python
# ❌ 数组查找 O(n)，总复杂度 O(n²)
if num - 1 in nums:  # 每次 in 遍历整个数组

# ✅ 集合查找 O(1)，总复杂度 O(n)
s = set(nums)
if num - 1 not in s:  # 哈希查找，常数时间
```

### 点 2：起点判断是核心

```python
if num - 1 not in s:
    # num 是起点——前面没数了，从这里开始数
```

没有这一行，`[1,2,3,4]` 会被从 1、2、3 分别扫描，每个数被数多次。有这一行，**只有 1 是起点**，2、3、4 在 when 循环里被「顺带」数过，外层遍历到它们时直接跳过。

### 点 3：嵌套循环 ≠ O(n²)

不是看到嵌套就判定 O(n²)。关键看**每个元素被内层循环访问的次数**：

```
序列 [1,2,3,4] [100,101] [200]
起点：1（while 数了 2,3,4）、100（while 数了 101）、200（没有下一个）
总 while 执行次数：3 + 1 + 0 = 4 次
外层 for 遍历 7 个元素
总操作 ≤ 7 + 4 = 11 < 2×7 ← O(n)
```

**每个数字在整个算法生命周期中，最多被 while 访问一次。各序列互不重叠。**

### 点 4：去重不影响结果

`[1,0,1,2]` → `set = {0,1,2}`，最长连续 `[0,1,2]` 长度 3。重复数字被集合自动过滤，对「连续」的判断无影响——`1,2` 是连续，`1,1,2` 也是连续，长度不变量。

### 点 5：return 必须放 for 外面

```python
# ❌ 放在 if 里 → 第一条链结束就返回，后面更长的链被忽略
if num - 1 not in s:
    ...
    return longest

# ❌ 放在 for 里（与 if 平级）→ 第一次遍历就返回
    return longest

# ✅ 放在 for 外面 → 遍历完所有起点再返回
return longest
```

### 点 6：longest 更新放 if 内还是外？

放 if 内（当前写法）：只有发现起点并数完一条链后，才有新长度需要比较。逻辑干净，避免非起点元素做无意义比较。放 if 外也可以，但每次循环都多一次 `max` 调用——开销可忽略，但意图不够清晰。

## 落到我身上

- **今天能做的**：下次遇到「要求 O(n) 但有嵌套循环」的题目，不急着判断 O(n²)——画一张「每个元素被访问次数」的表，确认内层循环不会重复访问同一元素
- **面试防坑**：这道题面试官最爱问「嵌套循环为什么不是 O(n²)」——答案不是背「每个数只被访问一次」，而是**画图证明**各连续序列互不重叠
- **编码习惯**：`in` 操作写之前先想容器类型——数组 O(n)，集合/字典 O(1)，选错容器 O(n) 变 O(n²)
